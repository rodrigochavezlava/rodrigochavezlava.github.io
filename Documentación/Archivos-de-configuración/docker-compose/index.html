Docker-Compose.yml¶

El archivo docker-compose.yml contiene la configuración de varios servicios que forman parte del sistema. Para facilitar la comprensión, explicaremos cada servicio por separado.

Comenzando con el servicio de Elasticsearch: Es importante destacar que todos los servicios utilizan la misma versión, en este caso la 7.15.2. Es crucial tener la misma versión en un sistema con servicios interconectados para evitar incompatibilidades y fallos de seguridad.

services:
  elasticsearch:
    image: docker.elastic.co/elasticsearch/elasticsearch:7.15.2

Dentro del archivo de Docker Compose, el servicio de Elasticsearch es fundamental ya que todos los demás servicios dependen de él. Si Elasticsearch falla, todos los servicios que dependen de él dejarán de funcionar. Por esta razón, se verifica constantemente si Elasticsearch está operativo. En caso de detectar un fallo, los demás servicios se detendrán y esperarán a que Elasticsearch se recupere. La configuración de Elasticsearch es más precisa que la de otros servicios. 

depends_on:
      - elasticsearch

En este caso, se encuentra en el modo "single-node", lo que significa que no hay más nodos y Elasticsearch funciona de forma independiente y segura.

- "discovery.type=single-node"      # Tipo de descubrimiento para Elasticsearch

Pasando al servicio de Kibana: Este servicio es más independiente y no afecta directamente a los demás servicios. Si solo se produce un fallo en Kibana, los demás servicios seguirán funcionando normalmente. Kibana utiliza el puerto predeterminado y se conecta a Elasticsearch para recibir información y datos.

environment:
      ELASTICSEARCH_URL: <http://elasticsearch:9200>  # URL de Elasticsearch

En cuanto al servicio de Filebeat: Este servicio es un emisor que centraliza los logs de diferentes ubicaciones. Es importante resaltar que se configura con permisos de solo lectura (:ro) para mayor seguridad. Al no requerir escritura, se reduce un punto de vulnerabilidad en el sistema 

- ./filebeat.yml:/usr/share/filebeat/filebeat.yml:ro

El servicio de Logstash: se encarga de procesar y enriquecer los datos de los eventos antes de enviarlos a Elasticsearch. Realiza tareas como la conversión y eliminación de datos críticos o sensibles. Logstash utiliza su puerto por defecto y también depende de Elasticsearch para enviar los datos procesados..

Por otro lado, el servicio de Suricata es una herramienta de detección de amenazas y prevención de intrusiones en red. Su configuración es ligeramente diferente, ya que vigila constantemente la red y requiere algunas configuraciones adicionales. También se configura como de solo lectura (:ro) debido a que es un sistema de eventos en tiempo real. Vigila las amenazas y detecta cualquier actividad maliciosa para minimizar el daño posible. Se especifica la interfaz a la que Suricata estará atento y se pueden configurar diferentes opciones según las necesidades.

Finalmente, el archivo de Docker Compose también incluye la versión del mismo y los volúmenes necesarios para almacenar los datos de Elasticsearch

 - ./suricata.yaml:/etc/suricata/suricata.yaml:ro       # Archivo de configuración de Suricata
    network_mode: "host"
    command: -c /etc/suricata/suricata.yaml -i eth0     # Comando para ejecutar Suricata

También hay que especificar la interfaz que vigila y se puede especificar con distitnas opciones.

version: '3.7'  # Versión de la sintaxis del archivo docker-compose

services:
  elasticsearch:  # Servicio de Elasticsearch
    image: docker.elastic.co/elasticsearch/elasticsearch:7.15.2  # Imagen utilizada para el contenedor
    container_name: elasticsearch  # Nombre del contenedor
    environment:
      - node.name=elasticsearch  # Configuración del nombre del nodo Elasticsearch
      - cluster.name=es-docker-cluster  # Configuración del nombre del clúster Elasticsearch
      - bootstrap.memory_lock=true  # Configuración para bloquear la memoria
      - "ES_JAVA_OPTS=-Xms512m -Xmx512m"  # Opciones de configuración para la memoria de Elasticsearch
      - "discovery.type=single-node"  # Configuración del tipo de descubrimiento del nodo Elasticsearch
      - "xpack.security.enabled=false"  # Configuración para deshabilitar la seguridad de X-Pack
    ulimits:
      memlock:
        soft: -1  # Límite blando para bloqueo de memoria
        hard: -1  # Límite duro para bloqueo de memoria
    volumes:
      - esdata:/usr/share/elasticsearch/data  # Volumen utilizado para almacenar los datos de Elasticsearch
    ports:
      - 9200:9200  # Mapeo del puerto del contenedor al puerto del host

  kibana:  # Servicio de Kibana
    image: docker.elastic.co/kibana/kibana:7.15.2  # Imagen utilizada para el contenedor
    container_name: kibana  # Nombre del contenedor
    ports:
      - 5601:5601  # Mapeo del puerto del contenedor al puerto del host
    environment:
      ELASTICSEARCH_URL: http://elasticsearch:9200/  # URL de Elasticsearch
    volumes:
      - ./kibana.yml:/usr/share/kibana/config/kibana.yml  # Volumen utilizado para la configuración de Kibana
    depends_on:
      - elasticsearch  # Dependencia del servicio de Elasticsearch

  filebeat:  # Servicio de Filebeat
    image: docker.elastic.co/beats/filebeat:7.15.2  # Imagen utilizada para el contenedor
    container_name: filebeat  # Nombre del contenedor
    user: root
    volumes:
      - ./filebeat.yml:/usr/share/filebeat/filebeat.yml:ro  # Volumen utilizado para la configuración de Filebeat
      - /var/log:/var/log:ro  # Mapeo del directorio de logs del host al contenedor
    depends_on:
      - elasticsearch  # Dependencia del servicio de Elasticsearch

  logstash:  # Servicio de Logstash
    image: docker.elastic.co/logstash/logstash:7.15.2  # Imagen utilizada para el contenedor
    container_name: logstash  # Nombre del contenedor
    volumes:
      - ./logstash.conf:/usr/share/logstash/pipeline/logstash.conf:ro  # Volumen utilizado para la configuración de Logstash
    ports:
      - 5044:5044  # Mapeo del puerto del contenedor al puerto del host
    depends_on:
      - elasticsearch  # Dependencia del servicio de Elasticsearch

  suricata:  # Servicio de Suricata
    image: jasonish/suricata:latest  # Imagen utilizada para el contenedor
    user: root
    privileged: true
    volumes:
      - /var/log/suricata:/var/log/suricata  # Mapeo del directorio de logs del host al contenedor
      - ./suricata.yaml:/etc/suricata/suricata.yaml:ro  # Volumen utilizado para la configuración de Suricata
      - /var/lib/suricata/rules:/var/lib/suricata/rules  # Mapeo del directorio de reglas del host al contenedor
      - /etc/suricata/classification.config:/etc/suricata/classification.config  # Mapeo del archivo de clasificación del host al contenedor
    network_mode: "host"  # Modo de red utilizado (comparte la red del host)
    command: -c /etc/suricata/suricata.yaml -i enp0s3  # Comando a ejecutar dentro del contenedor de Suricata

volumes:
  esdata:
    driver: local  # Tipo de driver de volumen utilizado (local)

networks:
  default:
    external:
      name: encriptadisimo  # Nombre de la red externa utilizada
